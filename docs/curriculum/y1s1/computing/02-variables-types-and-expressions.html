01 · Types: what a value is

Every value in Python has a type, and the type decides what operations mean. Mechanics uses four constantly: int — whole numbers, exact; float — decimals, approximate by nature; str — text, including text that LOOKS like a number; and bool — True or False, the currency of decisions. The classic trap sits between str and float: data arrives from files and instruments as text, and "42.5" will not do arithmetic — "42.5" * 2 is not 85. Conversion is explicit: float("42.5") crosses the border, and crossing it consciously is the habit.

02 · Floats tell the truth approximately

A float is a binary approximation of a decimal number, and the approximation shows: \( 0.1 + 0.2 \) is not exactly \( 0.3 \) in a computer, and testing equality between computed floats fails unpredictably. The professional pattern is the tolerance test — \( \texttt{abs(a - b) < 1e-9} \) — precisely the pattern every quiz verifier on this platform uses. Roundoff also accumulates: sum a million small floats and the last digits drift. None of this is a defect to fix; it is a property to respect, the numerical equivalent of instrument uncertainty.

03 · Presenting numbers like an engineer

Computed values arrive with fifteen digits; engineering answers carry units and honest precision. The f-string is the tool: f"{p/1000:.2f} kW" renders 8175 W as \( \texttt{8.18 kW} \) — converted, rounded, and labeled in one expression. Precision in the format is a claim about accuracy, exactly as significant figures were in the physics course: print three decimals only if the calculation deserves them. The output line of a script is its public face; make it state value, precision, and unit together, every time.

04 · The type-discipline working method

Treat every value's type as consciously as its unit. At the borders of the script — reading files, taking input — convert text to numbers explicitly and immediately, so the interior works entirely in numeric types. Compare floats only with tolerances; reserve equality for ints and strings. Name booleans as assertions — is_running, above_alarm — so conditions read as sentences. And when a result looks absurd, check types before logic: half the beginner's "impossible" outputs are a string where a number belonged, silently concatenated instead of added. The interpreter always does exactly what the types tell it to; the working method is making sure that is also what you meant.

Keep the int/float division of labor deliberate as well. Counts, indices, and loop variables are integers by nature — a plant has 14 pumps, never 14.0 — while every measured quantity is a float carrying its approximation honestly. Mixing them is legal (Python promotes to float), but REPORTING a count as a decimal reads as sloppiness, and rounding a float into an index without deciding between truncation and rounding is how off-by-one errors are born. When precision genuinely matters — currency totals, long accumulations — know that specialized decimal types exist and that the ordinary float is a measuring instrument, not a ledger. For this course's engineering arithmetic, floats with tolerance comparisons and honest formatting carry every calculation; the discipline is knowing which type each value IS, the same way the physics course always knew each quantity's unit — a habit checked once per variable and repaid on every line after.

Examples and Quiz

Work every problem on paper first. Solved examples reveal their full solution; for the multiple-choice questions, select your answers and press Submit to see what was right, what was not, and why.

Problem 1 of 8 · solve, then check

The equality that fails

Problem. Predict the output of both lines, and write the correct comparison:

print(0.1 + 0.2 == 0.3)
print(abs((0.1 + 0.2) - 0.3) < 1e-9)

Solution. (1) First line: False — in binary, 0.1 and 0.2 are repeating fractions; their sum is 0.30000000000000004. (2) Second line: True — the difference (≈\( 5.5\times10^{-17} \)) is far inside the \( 10^{-9} \) tolerance. (3) The tolerance test IS the correct float comparison.

Answer: False, then True. Sanity check: this is not a Python defect — every binary floating-point system on earth does it. Engineering already knows the answer: no two measured values are ever "equal" either, only equal within tolerance; the code pattern just says so explicitly.

Problem 2 of 8 · solve, then check

Crossing the text border

Problem. A level reading arrives from a file as the text "42.5". Show what "42.5" * 2 produces, and script the correct doubling.

Solution. (1) "42.5" * 2 → "42.542.5" — string repetition, perfectly legal, silently wrong. (2) Correct: reading = float("42.5"); doubled = reading * 2 → 85.0.

Answer: text repetition vs the numeric 85.0 — the conversion makes the difference. Sanity check: the wrong version raises no error at all — the type system did exactly what strings do. Every file-reading script in this course will convert at the border for exactly this reason; the habit costs one function call and prevents the silent classic.

Problem 3 of 8 · solve, then check

Publishing a result properly

Problem. A script computes p = 8175 (watts). Write the output line that reports it in kW to two decimals with its unit — and state exactly what prints.

Solution. (1) print(f"shaft power = {p/1000:.2f} kW"). (2) \( 8175/1000 = 8.175 \); format \( \texttt{.2f} \) rounds to two decimals. (3) Prints: shaft power = 8.18 kW.

Answer: shaft power = 8.18 kW. Sanity check: conversion, rounding, and labeling happened in ONE expression — nothing was stored rounded (the full 8175 remains available for further computation), which is the right division of labor: full precision inside, honest precision at the interface.

Problem 4 of 8 · multiple choice

type("3.14") reports:

Quotes make it text, whatever it resembles: "3.14" is a string of four characters. It becomes a number only through explicit conversion — float("3.14"). Instrument logs, CSV files, and user input all deliver strings; treating them as numbers without conversion is the most common silent error in beginner data scripts.

Problem 5 of 8 · multiple choice

int(7.9) evaluates to:

int() truncates toward zero — it drops the fraction, it does not round: 7. Rounding is a different, explicit request: round(7.9) gives 8. The distinction bites in bin-counting and indexing arithmetic, where a truncation assumed to be a rounding shifts everything by one. Choose the operation by name, not by hope.

Problem 6 of 8 · multiple choice

Which comparison between two COMPUTED float values is professionally correct?

Computed floats carry binary roundoff, so equality is a coin toss; the tolerance test states how close is close enough — an engineering judgment made explicit. String comparison inherits the same digits problem in costume, and is tests object identity, a different concept entirely. Every verifier on this platform uses the tolerance pattern; now you know why.

Problem 7 of 8 · multiple choice

10 / 4 evaluates to:

Single slash is TRUE division and always returns a float: 2.5. Floor division 10 // 4 gives 2. (Older languages divided integers integrally, and imported habits from them cause exactly this surprise in reverse.) Knowing which division you asked for is part of type discipline — the operators are unambiguous even when memory isn't.

Problem 8 of 8 · multiple choice

A boolean variable controlling an alarm branch is best named:

Booleans are assertions, and naming them as assertions makes conditions read as sentences: if above_alarm: needs no comment. Generic names force every reader back up the script to discover what is being tested. (temp is doubly cursed — ambiguous between 'temporary' and 'temperature'.) Readable conditions are cheap insurance in any script an auditor may one day read.
