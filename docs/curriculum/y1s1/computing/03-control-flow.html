Control flow

Conditionals and loops for case logic and iteration. Guard clauses for invalid inputs. Translating a calculation procedure into control flow.

Lecture

After this lesson you can answer:
  • Turn 'if the reading exceeds limit, flag it' into control flow — what guards the invalid-input case?
  • When is a while loop honest and a for loop not?

01 · Decisions: if, elif, else

Programs earn their keep when behavior depends on data, and the instrument is the conditional. An if tests a boolean expression; elif chains further tests in order; else catches the remainder. Order matters and is a design decision: an alarm classifier tests the most severe threshold first, so a reading of 95 °C reports "trip" rather than the also-true "warning". Comparisons (\( \texttt{< <= == >= >} \)) and the boolean connectives (\( \texttt{and, or, not} \)) compose the tests; indentation — four spaces — is not decoration but syntax, defining exactly which lines each branch owns.

02 · Repetition: for and while

The for loop walks a known collection — every reading in a list, every line in a file, every integer in range(n) (which yields 0 through \( n-1 \)) — and is the workhorse of data processing: initialize an accumulator, add each item, report. The while loop repeats as long as a condition holds and serves the open-ended questions: keep compounding the filter's pressure drop until it crosses the alarm line, and report how many weeks that took. Choose by the question's shape — known collection, for; unknown horizon, while — and give every while a guaranteed exit, because a condition that never turns false runs forever.

03 · Guard clauses: refusing bad input early

Real calculations meet impossible inputs — negative flows, zero efficiencies, text where numbers belong — and the professional pattern rejects them at the door: if eta <= 0: raise ValueError("efficiency must be positive"). The guard clause converts a silent wrong answer into a loud, named failure at the exact line where the assumption broke. The mathematics course called this domain discipline — functions have domains, and \( \eta = 0 \) is not in this one; the guard writes the domain into the code.

04 · The control-flow working method

Translate procedures the way you would explain them aloud. Write the decision table first — conditions in severity order, outcomes named — then transcribe it as an if/elif/else chain and test EVERY branch once, including the else. For loops, name the accumulator for what it accumulates (total_kwh, not s), initialize it visibly, and hand-trace the first two iterations on paper before trusting a thousand — the two-minute trace that catches off-by-one and wrong-order errors while they are still cheap. Guard first, compute second, report third: scripts organized in that order read as procedures, fail loudly at their borders, and hold up when the data turns hostile, which production data always eventually does.

Two loop hygienes complete the method. Keep the loop body honest about what it touches — a loop that both accumulates a total AND mutates the list it walks is two jobs tangled, and tangled jobs breed the subtle bugs; split them. And when a while-loop's exit depends on a computed quantity, add a hard iteration cap as a guard — if weeks > 1000: raise RuntimeError(...) — so a modeling mistake produces a named failure in milliseconds instead of a silent forever. Loops are where scripts spend their lives; discipline there is discipline everywhere.

Examples and Quiz

Work every problem on paper first. Solved examples reveal their full solution; for the multiple-choice questions, select your answers and press Submit to see what was right, what was not, and why.

Problem 1 of 8 · solve, then check

Trace the alarm classifier

Problem. Trace this classifier for readings of 85, then 96 (°C):

if t >= 95:
    level = "TRIP"
elif t >= 80:
    level = "ALARM"
elif t >= 70:
    level = "WARNING"
else:
    level = "NORMAL"

Solution. (1) t = 85: the 95-test fails, the 80-test passes → "ALARM"; the 70-test is never reached. (2) t = 96: the FIRST test passes → "TRIP", chain exits immediately.

Answer: 85 → ALARM; 96 → TRIP. Sanity check: reverse the order — test 70 first — and a 96 °C reading would report "WARNING", since 96 ≥ 70 is also true and the first true branch wins. Severity-descending order is not style; it is the logic. This is why the working method writes the decision table before the code.

Problem 2 of 8 · solve, then check

The accumulator, traced

Problem. Trace the total across the loop:

readings = [4, 7, 2, 9]
total = 0
for r in readings:
    total = total + r

Solution. (1) total starts 0. (2) After each pass: 0+4=4; 4+7=11; 11+2=13; 13+9=22. (3) Final: total = 22.

Answer: 22. Sanity check: hand-tracing the first two iterations (4, then 11) is the working method's cheap audit — if those match expectation, the pattern is sound for four items or four million. Forgetting the initialization line, the classic slip, would crash on the first pass rather than silently corrupt: Python's favor to beginners.

Problem 3 of 8 · solve, then check

How many weeks to the alarm?

Problem. A filter's ΔP starts at 0.30 bar and compounds 5% weekly; the alarm is 0.60 bar. Write the while loop that counts the weeks, and state its result.

Solution. (1) dp = 0.30; weeks = 0 while dp < 0.60: weeks += 1 dp = 0.30 * 1.05 ** weeks (2) Week 14: \( 0.30 \times 1.05^{14} = 0.594 \) — still under. Week 15: \( 0.624 \) — crossed. (3) Result: weeks = 15.

Answer: 15 weeks. Sanity check: the mathematics course's logarithm gave 14.2 weeks for the exact crossing — the loop answers the OPERATIONAL question (first whole week past the line), hence 15. Same physics, two honest answers to two subtly different questions; knowing which one you asked is the engineering.

Problem 4 of 8 · multiple choice

range(5) iterates over:

range(n) starts at 0 and stops BEFORE n: five values, 0 through 4. The zero-start convention aligns with list indexing (Lesson 6) so range(len(data)) walks every index — and the off-by-one instincts imported from 1-counting are the classic loop bug this fact retires. When 1 through 5 is genuinely wanted: range(1, 6).

Problem 5 of 8 · multiple choice

The difference between = and == is:

x = 5 stores; x == 5 asks. Python refuses plain assignment inside an if-condition, converting the classic typo into a visible syntax error — a kindness not all languages extend. Reading them aloud differently ('gets' versus 'equals') keeps the two straight in review, where the eye alone glides past a single missing character.

Problem 6 of 8 · multiple choice

A guard clause like if eta <= 0: raise ValueError(...) exists to:

Fed η = 0, the unguarded script divides by zero somewhere downstream — or worse, a negative η produces a plausible-looking negative power that travels into a report. The guard names the broken assumption at the exact border where it entered. It is the code form of a domain statement, and auditors read guards as the script's declared operating envelope.

Problem 7 of 8 · multiple choice

After total = 0 and for i in range(3): total += i, total equals:

The loop adds 0, 1, 2 — sum 3. Answering 6 adds 1+2+3, the off-by-one imported from 1-counting; range's zero-start (previous question) decides it. The two-iteration hand trace — total: 0, then 1, then 3 — settles such questions in seconds, which is why the working method makes it a reflex before any loop is trusted at scale.

Problem 8 of 8 · multiple choice

Inside a loop scanning readings for the first value above a limit, the statement that exits immediately upon finding it is:

break leaves the loop at once — the found-it exit that saves scanning the rest. (continue, its sibling, skips to the next iteration instead.) There is no stop; killing the interpreter to end a loop is surgery with a hammer. Early exits also document intent: a reader sees the loop is a SEARCH, not a full sweep.
