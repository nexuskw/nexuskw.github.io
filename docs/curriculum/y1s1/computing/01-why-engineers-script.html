Why engineers script

The spreadsheet's limits and the script's advantages: repeatability, audit trail, scale. Setting up Python and running a first calculation. Version thinking for engineering work.

01 · Why engineers script

A spreadsheet hides its logic inside cells; a script lays it out as readable text. That single difference buys the three things engineering calculation needs most: repeatability — run the script tomorrow on new data and the identical logic executes; auditability — a reviewer reads the calculation top to bottom, every assumption visible; and scale — the loop that processes one reading processes a million without new effort. Spreadsheets remain excellent scratchpads, but the moment a calculation matters, recurs, or grows, the professional tool is a script — and the habit of writing one is what this course installs.

02 · The toolchain, plainly

Python is an interpreter: it reads statements and executes them immediately. You can feed it lines interactively — the REPL, ideal for trying an idea — or hand it a file of statements, a script, which is the durable product. A script is just text: openable in any editor, storable beside the data it processed, and — crucially — diffable, so yesterday's version and today's can be compared line by line. That property makes scripts the natural citizens of version control and the natural evidence in an engineering audit trail.

03 · Anatomy of a calculation script

Every calculation script worth keeping has the same skeleton: named inputs at the top, with units in the names or comments (\( \texttt{flow\_m3h = 100} \)); the computation in clear steps, mirroring how you would write it on paper; and output that states units — \( \texttt{print(f"shaft power = \{P/1000:.1f\} kW")} \). Arithmetic operators behave as expected, with three worth flagging now: \( \texttt{**} \) is exponentiation, \( \texttt{/} \) always returns a decimal, and \( \texttt{//} \) divides and discards the remainder. A pump-power script in this shape is eight lines — and it replaces the back-of-envelope forever, because it IS the envelope, kept.

04 · The scripting working method

Write scripts the way the mathematics course taught you to check units: as a discipline, not an afterthought. Name variables with their units — ambiguity between bar and kPa has sunk more calculations than any syntax error ever will. State inputs once, at the top, never buried mid-logic. Print results WITH units and sensible precision. Keep one calculation per script while learning; later lessons modularize. And before trusting any script, run it against a case you have verified by hand — the machine-verification habit this whole platform practices on you is the habit you are now learning to practice on your own code. A script that has never met a hand-checked case is an opinion with syntax.

Finally, keep scripts with their evidence. Store the script beside the data it processed and note the date and source of both — the calculation, its inputs, and its result then travel as one reviewable package. Months later, when a result is questioned, the answer is a rerun rather than an archaeology project; and when the data changes, the same package regenerates every downstream number identically. That closed loop — data, script, result, rerunnable — is what engineering computing means by an audit trail.

Problem 1 of 8 · solve, then check

Trace the pump-power script

Problem. Trace this script and state its output:

rho = 1000        # kg/m3
g = 9.81          # m/s2
flow_m3h = 100
head_m = 30
eta = 0.70
Q = flow_m3h / 3600
P = rho * g * Q * head_m / eta
print(f"shaft power = {P/1000:.1f} kW")

Solution. (1) \( Q = 100/3600 = 0.02778\ \mathrm{m^3/s} \). (2) \( P = 1000 \times 9.81 \times 0.02778 \times 30 / 0.70 = 11{,}679\ \mathrm{W} \). (3) Output: shaft power = 11.7 kW.

Answer: 11.7 kW. Sanity check: this is the exact calculation the mathematics course verified by hand — hydraulic 8.2 kW, divided by 0.7. The script is that envelope, kept: change the flow tomorrow and the identical logic reruns, which is the entire argument of 1.

Problem 2 of 8 · solve, then check

A conversion worth scripting

Problem. Write (and trace for 6.0 barg) a script converting gauge pressure in bar to absolute pressure in kPa: \( P_{abs} = P_{gauge} \times 100 + 101.325 \).

Solution. (1) p_barg = 6.0; p_kpa_abs = p_barg * 100 + 101.325; print(f"{p_kpa_abs:.1f} kPa abs"). (2) Trace: \( 6.0 \times 100 + 101.325 = 701.3\ \mathrm{kPa\ abs} \).

Answer: 701.3 kPa absolute. Sanity check: the gauge trap from the mathematics course, now armored in code: the script's NAME says barg in and kPa-absolute out, so the assumption travels with the calculation — which is precisely what a cell formula never tells its reader.

Problem 3 of 8 · solve, then check

The running-cost one-liner

Problem. A 15 kW machine runs 16 h/day at an energy price of 0.045 per kWh. Script the daily cost and trace it.

Solution. (1) cost = 15 * 16 * 0.045. (2) \( 15 \times 16 = 240\ \mathrm{kWh/day} \); \( 240 \times 0.045 = 10.8 \) per day.

Answer: 10.8 currency units per day — ≈3,940 per year of continuous duty. Sanity check: kW times hours gives kWh (energy), times price-per-kWh gives money — the units chain cleanly, and the annualized figure (×365) is the number that turns a motor-efficiency conversation into a business case. Three inputs, one line, permanent tool.

Problem 4 of 8 · multiple choice

The decisive advantage of a script over a spreadsheet for a recurring engineering calculation is:

Auditability and repeatability are the engineering virtues: every step readable top to bottom, every rerun identical. Scripts absolutely can contain errors — the hand-check rule exists because of it; speed rarely matters at calculation scale; and spreadsheets compute fine — they just hide the logic in cells where reviewers cannot see it whole.

Problem 5 of 8 · multiple choice

In Python, 7 // 2 evaluates to:

// is floor division: divide and discard the remainder — 3. Plain / gives 3.5 (always a float); rounding UP would give 4 (that's math.ceil); 1 is the REMAINDER, delivered by %. The three operators split the division jobs among themselves, and mixing them up is the first classic Python slip.

Problem 6 of 8 · multiple choice

print(2 ** 3) outputs:

** is exponentiation: \( 2^3 = 8 \). The caret ^ — which many calculators use — means something entirely different in Python (bitwise XOR), and 2^3 would print 1: a legal expression computing the wrong thing silently, which is exactly the kind of error the hand-check rule is designed to catch.

Problem 7 of 8 · multiple choice

The best variable name for a flow reading of 100 m³/h is:

The unit rides in the name, so every later line that touches it knows what it holds — the script equivalent of the mathematics course's unit discipline. Single letters and generic names create the bar-versus-kPa ambiguities that sink calculations; six months later, flow_m3h still explains itself and x1 explains nothing.

Problem 8 of 8 · multiple choice

Before a new calculation script is trusted, it should first:

Running without errors proves only that the syntax is legal — a script can be fluently, confidently wrong. The hand-checked case pins the LOGIC: if the script reproduces a known answer, its machinery is sound and may be pointed at unknown cases. Repetition of an unverified script just repeats the doubt. This is the platform's own machine-verification habit, handed to you.
