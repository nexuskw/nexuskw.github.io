Testing and debugging engineering code

Assertions and test cases that pin a calculation to a hand-checked answer before it is trusted at scale. Exceptions and guard clauses for bad inputs; unit sanity checks as an executable habit. Debugging as hypothesis testing, not stare-and-hope.

After this lesson you can answer:

01 · Why engineers test code

This platform machine-verifies every numeric answer before publishing it; this lesson hands you the same discipline for your own scripts. A test is a claim made executable: run the function on a case whose answer you know, and let the computer confirm agreement — today, tomorrow, and after every future edit. Untested code decays silently: an innocent change to one line shifts a result three functions away, and without tests nobody notices until the wrong number is in a report. With tests, the same mistake fails loudly in seconds, at your desk, before it costs anything.

02 · The assertion: a claim with teeth

The core tool is one statement: assert condition, "message" — silent when the condition holds, a named failure when it breaks. For engineering functions the pattern is the tolerance form from Lesson 2: assert abs(area_m2(0.1) - 0.007854) < 1e-6 — the hand-checked case (a 100 mm circle's \( \pi d^2/4 \)) pinned beside the function it certifies. Three or four assertions per function — a typical case, an edge (zero, the domain boundary), and a case exercising each branch — form a small permanent test suite that reruns in milliseconds. The habit converts "I checked it once when I wrote it" into "it is checked on every run", which is the difference between a memory and a guarantee.

03 · Exceptions and honest failure

Guard clauses (Lesson 3) raise exceptions at broken assumptions; testing meets them from the other side. A function fed garbage should fail LOUDLY — ValueError with a message naming the violated assumption — never return a plausible wrong number. Debugging itself is hypothesis testing, the scientific method at a keyboard: reproduce the failure with the smallest possible input, form a hypothesis about the earliest wrong value, place a print or assertion just before the suspect line, and bisect until the bug has nowhere to hide. Staring is not a method; bisection is.

04 · The verification working method

Adopt the platform's own pipeline at personal scale. Write the function; write its hand-checked case as an assertion immediately — while the paper calculation is still on the desk; add the edge-case and branch assertions; keep the whole block at the bottom of the file or in a test script, and run it after EVERY change. When a bug surfaces in use, its fix begins by writing the failing case as a new assertion — so the same bug can never return unannounced. And respect the honest limit: tests prove the presence of agreement with your checked cases, not the absence of all error — which is precisely why the cases are chosen with engineering judgment: the typical, the extreme, and the branch nobody believes will ever run. That judgment is the engineering; the assertions merely keep it enforced. A dozen well-chosen assertions guarding a personal library of checked functions is a small thing to maintain and a large thing to own: it is the difference between code you wrote and code you can vouch for.

Problem 1 of 8 · solve, then check

Pin the area function

Problem. Given def area_m2(d_m): return 3.14159 * d_m**2 / 4, write the tolerance assertion pinning it to the hand-checked case d = 0.1 m, and state whether it passes.

Solution. (1) Hand check: \( A = \pi d^2/4 = \pi \times 0.01/4 = 0.007854\ \mathrm{m^2} \). (2) Assertion: assert abs(area_m2(0.1) - 0.007854) < 1e-6. (3) The function's 5-digit π gives 0.0078540 — inside tolerance: PASSES.

Answer: the assertion passes; the function is pinned. Sanity check: the tolerance \( 10^{-6} \) was chosen to admit the 5-digit π but reject real errors — a factor-of-4 slip or a missing square lands orders of magnitude away. Tolerances are engineering judgments: tight enough to catch bugs, loose enough to ignore roundoff.

Problem 2 of 8 · solve, then check

Find the bug by its units

Problem. This function claims to return lifting power in watts, but its test fails. Locate the bug and fix it:

def lift_power_w(mass_kg, height_m, time_s):
    """Average power to lift mass through height in given time."""
    return mass_kg * 9.81 * height_m

assert abs(lift_power_w(400, 12, 60) - 784.8) < 0.1

Solution. (1) The return line computes \( mgh \) — ENERGY (47,088 J), never dividing by time: the units are joules, not watts. (2) Fix: return mass_kg * 9.81 * height_m / time_s. (3) Check: \( 400 \times 9.81 \times 12/60 = 784.8\ \mathrm{W} \) — the assertion passes.

Answer: missing / time_s; corrected, the test passes. Sanity check: the physics course's energy-versus-power distinction caught a software bug — the failing assertion pointed at it, and the unit audit named it. This is the standard rhythm: the test detects, the engineering diagnoses.

Problem 3 of 8 · solve, then check

Guard, then prove the guard

Problem. Extend shaft_power with a guard against non-positive efficiency, and write the test that PROVES the guard fires.

Solution. (1) Guard: if eta <= 0: raise ValueError("eta must be positive") as the function's first line. (2) Proving it: try: call with eta=0; if no exception arrives, assert False, "guard did not fire"; in the except ValueError: branch, pass. (3) Now BOTH behaviors are pinned: right answers for good input, loud refusal for bad.

Answer: guard + try/except test, both permanent. Sanity check: testing that failure HAPPENS feels odd once — then obvious: the guard is a promised behavior like any other, and an accidental deletion of it should fail the suite. Safety systems are tested by attempting what they must prevent; code guards earn the same respect.

Problem 4 of 8 · multiple choice

An assertion beside a function buys you, above a one-time manual check:

The manual check evaporates the moment it scrolls off screen; the assertion re-executes it forever — especially after edits, when yesterday's correctness is most quietly at risk. Tests never prove the absence of all error (the cases are finite), and they cost lines and microseconds — a spectacular exchange rate for the guarantee they do provide.

Problem 5 of 8 · multiple choice

The most valuable additional test cases for flow_velocity(q, d) beyond the typical case are:

Functions break at their edges: zero triggers divisions and log domains, extremes expose unit and overflow assumptions. Five similar mid-range cases re-test the same path five times; edges test the paths that fail in production at 3 a.m. Random inputs without known answers verify nothing — a test needs an expected value to compare against.

Problem 6 of 8 · multiple choice

Fed an impossible input, a professional calculation function should:

A returned 0 or guess is a plausible number that travels — into comparisons, reports, decisions — with nothing marking it poisoned. The exception stops the pipeline at the exact border where the assumption broke, with a message naming it. Printed warnings scroll past unread in any automated run. Loud early failure is cheap; quiet wrong numbers compound.

Problem 7 of 8 · multiple choice

The systematic way to locate a bug in a 40-line calculation is:

Bisection: probe the middle — if values are right there, the bug is downstream; wrong, upstream. Each probe halves the search; 40 lines fall in ~6 probes. Rereading finds what you expect to see (which is how the bug survived writing); rewriting risks a new bug and loses the lesson; blanket try/except SILENCES failures — the opposite of locating them.

Problem 8 of 8 · multiple choice

After fixing a bug, the professional final step is:

The regression test: the input that exposed the bug becomes a permanent claim, so the same failure can never return unannounced — including via some future 'improvement' that reintroduces it. Commented-out code is noise that rots; 'it's fixed' is true only until the next edit. Suites grow by exactly this route: every scar becomes armor.
