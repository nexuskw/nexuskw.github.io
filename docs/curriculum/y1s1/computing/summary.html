Part 1 — Lectures

Lesson 01 · Why engineers script

01 · Why engineers script

A spreadsheet hides its logic inside cells; a script lays it out as readable text. That single difference buys the three things engineering calculation needs most: repeatability — run the script tomorrow on new data and the identical logic executes; auditability — a reviewer reads the calculation top to bottom, every assumption visible; and scale — the loop that processes one reading processes a million without new effort. Spreadsheets remain excellent scratchpads, but the moment a calculation matters, recurs, or grows, the professional tool is a script — and the habit of writing one is what this course installs.

02 · The toolchain, plainly

Python is an interpreter: it reads statements and executes them immediately. You can feed it lines interactively — the REPL, ideal for trying an idea — or hand it a file of statements, a script, which is the durable product. A script is just text: openable in any editor, storable beside the data it processed, and — crucially — diffable, so yesterday's version and today's can be compared line by line. That property makes scripts the natural citizens of version control and the natural evidence in an engineering audit trail.

03 · Anatomy of a calculation script

Every calculation script worth keeping has the same skeleton: named inputs at the top, with units in the names or comments (\( \texttt{flow\_m3h = 100} \)); the computation in clear steps, mirroring how you would write it on paper; and output that states units — \( \texttt{print(f"shaft power = \{P/1000:.1f\} kW")} \). Arithmetic operators behave as expected, with three worth flagging now: \( \texttt{**} \) is exponentiation, \( \texttt{/} \) always returns a decimal, and \( \texttt{//} \) divides and discards the remainder. A pump-power script in this shape is eight lines — and it replaces the back-of-envelope forever, because it IS the envelope, kept.

04 · The scripting working method

Write scripts the way the mathematics course taught you to check units: as a discipline, not an afterthought. Name variables with their units — ambiguity between bar and kPa has sunk more calculations than any syntax error ever will. State inputs once, at the top, never buried mid-logic. Print results WITH units and sensible precision. Keep one calculation per script while learning; later lessons modularize. And before trusting any script, run it against a case you have verified by hand — the machine-verification habit this whole platform practices on you is the habit you are now learning to practice on your own code. A script that has never met a hand-checked case is an opinion with syntax.

Finally, keep scripts with their evidence. Store the script beside the data it processed and note the date and source of both — the calculation, its inputs, and its result then travel as one reviewable package. Months later, when a result is questioned, the answer is a rerun rather than an archaeology project; and when the data changes, the same package regenerates every downstream number identically. That closed loop — data, script, result, rerunnable — is what engineering computing means by an audit trail.

Lesson 02 · Variables, types, and expressions

01 · Types: what a value is

Every value in Python has a type, and the type decides what operations mean. Mechanics uses four constantly: int — whole numbers, exact; float — decimals, approximate by nature; str — text, including text that LOOKS like a number; and bool — True or False, the currency of decisions. The classic trap sits between str and float: data arrives from files and instruments as text, and "42.5" will not do arithmetic — "42.5" * 2 is not 85. Conversion is explicit: float("42.5") crosses the border, and crossing it consciously is the habit.

02 · Floats tell the truth approximately

A float is a binary approximation of a decimal number, and the approximation shows: \( 0.1 + 0.2 \) is not exactly \( 0.3 \) in a computer, and testing equality between computed floats fails unpredictably. The professional pattern is the tolerance test — \( \texttt{abs(a - b) < 1e-9} \) — precisely the pattern every quiz verifier on this platform uses. Roundoff also accumulates: sum a million small floats and the last digits drift. None of this is a defect to fix; it is a property to respect, the numerical equivalent of instrument uncertainty.

03 · Presenting numbers like an engineer

Computed values arrive with fifteen digits; engineering answers carry units and honest precision. The f-string is the tool: f"{p/1000:.2f} kW" renders 8175 W as \( \texttt{8.18 kW} \) — converted, rounded, and labeled in one expression. Precision in the format is a claim about accuracy, exactly as significant figures were in the physics course: print three decimals only if the calculation deserves them. The output line of a script is its public face; make it state value, precision, and unit together, every time.

04 · The type-discipline working method

Treat every value's type as consciously as its unit. At the borders of the script — reading files, taking input — convert text to numbers explicitly and immediately, so the interior works entirely in numeric types. Compare floats only with tolerances; reserve equality for ints and strings. Name booleans as assertions — is_running, above_alarm — so conditions read as sentences. And when a result looks absurd, check types before logic: half the beginner's "impossible" outputs are a string where a number belonged, silently concatenated instead of added. The interpreter always does exactly what the types tell it to; the working method is making sure that is also what you meant.

Keep the int/float division of labor deliberate as well. Counts, indices, and loop variables are integers by nature — a plant has 14 pumps, never 14.0 — while every measured quantity is a float carrying its approximation honestly. Mixing them is legal (Python promotes to float), but REPORTING a count as a decimal reads as sloppiness, and rounding a float into an index without deciding between truncation and rounding is how off-by-one errors are born. When precision genuinely matters — currency totals, long accumulations — know that specialized decimal types exist and that the ordinary float is a measuring instrument, not a ledger. For this course's engineering arithmetic, floats with tolerance comparisons and honest formatting carry every calculation; the discipline is knowing which type each value IS, the same way the physics course always knew each quantity's unit — a habit checked once per variable and repaid on every line after.

Lesson 03 · Control flow

01 · Decisions: if, elif, else

Programs earn their keep when behavior depends on data, and the instrument is the conditional. An if tests a boolean expression; elif chains further tests in order; else catches the remainder. Order matters and is a design decision: an alarm classifier tests the most severe threshold first, so a reading of 95 °C reports "trip" rather than the also-true "warning". Comparisons (\( \texttt{< <= == >= >} \)) and the boolean connectives (\( \texttt{and, or, not} \)) compose the tests; indentation — four spaces — is not decoration but syntax, defining exactly which lines each branch owns.

02 · Repetition: for and while

The for loop walks a known collection — every reading in a list, every line in a file, every integer in range(n) (which yields 0 through \( n-1 \)) — and is the workhorse of data processing: initialize an accumulator, add each item, report. The while loop repeats as long as a condition holds and serves the open-ended questions: keep compounding the filter's pressure drop until it crosses the alarm line, and report how many weeks that took. Choose by the question's shape — known collection, for; unknown horizon, while — and give every while a guaranteed exit, because a condition that never turns false runs forever.

03 · Guard clauses: refusing bad input early

Real calculations meet impossible inputs — negative flows, zero efficiencies, text where numbers belong — and the professional pattern rejects them at the door: if eta <= 0: raise ValueError("efficiency must be positive"). The guard clause converts a silent wrong answer into a loud, named failure at the exact line where the assumption broke. The mathematics course called this domain discipline — functions have domains, and \( \eta = 0 \) is not in this one; the guard writes the domain into the code.

04 · The control-flow working method

Translate procedures the way you would explain them aloud. Write the decision table first — conditions in severity order, outcomes named — then transcribe it as an if/elif/else chain and test EVERY branch once, including the else. For loops, name the accumulator for what it accumulates (total_kwh, not s), initialize it visibly, and hand-trace the first two iterations on paper before trusting a thousand — the two-minute trace that catches off-by-one and wrong-order errors while they are still cheap. Guard first, compute second, report third: scripts organized in that order read as procedures, fail loudly at their borders, and hold up when the data turns hostile, which production data always eventually does.

Two loop hygienes complete the method. Keep the loop body honest about what it touches — a loop that both accumulates a total AND mutates the list it walks is two jobs tangled, and tangled jobs breed the subtle bugs; split them. And when a while-loop's exit depends on a computed quantity, add a hard iteration cap as a guard — if weeks > 1000: raise RuntimeError(...) — so a modeling mistake produces a named failure in milliseconds instead of a silent forever. Loops are where scripts spend their lives; discipline there is discipline everywhere.

Lesson 04 · Functions

01 · Functions: calculations with names

A function packages a calculation behind a name and an interface: def shaft_power(q_m3s, head_m, eta=0.7): declares what goes in, return declares what comes out, and everything between is the calculation, written once and reused forever. The docstring — a quoted paragraph directly under the def — states what the function does AND its units, which turns the function into a self-documenting engineering tool: help(shaft_power) answers the reviewer before the reviewer asks. Default arguments (the eta=0.7) encode the usual case while keeping the unusual case one keyword away.

02 · Return, not print

A function that PRINTS its result has spent it; a function that RETURNS it hands the value to the caller for further use — feeding the next calculation, a comparison, a plot, a test. Printing is presentation, the script's outer layer; returning is plumbing, the composable interior. The distinction becomes visible the first time you try to use a printed result in arithmetic and find you hold None. Compose functions the way the chain rule composed rates: small functions, each doing one nameable thing, wired by their returns — \( \texttt{gap(c\_to\_k(90), c\_to\_k(30))} \) reads like the sentence it implements.

03 · Scope: what happens in a function stays there

Variables born inside a function are local: invisible outside, gone when the call ends. This is a feature — ten functions can each use x without collision — and a discipline: a function should receive what it needs as arguments and deliver what it produces as returns, never reaching sideways into the script's globals. Functions written that way are testable in isolation, which is exactly how the next lesson will test them.

04 · The function-design working method

Extract a function whenever a calculation is used twice, or once but nameable. Name it by what it returns (hydraulic_power_w), carry units in parameter names, and write the docstring FIRST — if you cannot state the contract in a sentence, the function is not yet one thing. Keep functions small enough to hand-verify: one formula, one decision, one transformation each. Then apply Lesson 1's rule at the function scale: every function meets a hand-checked case before it is trusted, and the check lives beside the function as next lesson's assertion. A personal library of such functions — built, checked, and reused across this course — is the deliverable this working method is quietly assembling.

Resist the two classic function pathologies while the habits are forming. The do-everything function — read the file, convert the units, compute the power, print the report — cannot be tested except by running the world; split it at every nameable seam and each piece becomes checkable alone. And the copy-paste twin — the same formula pasted in three places with one variable renamed — guarantees that a future correction reaches two of the three; extraction into one function makes the fix land everywhere by construction. Functions are the unit of reuse, of testing, and of trust: size them so all three work.

Lesson 05 · Testing and debugging engineering code

01 · Why engineers test code

This platform machine-verifies every numeric answer before publishing it; this lesson hands you the same discipline for your own scripts. A test is a claim made executable: run the function on a case whose answer you know, and let the computer confirm agreement — today, tomorrow, and after every future edit. Untested code decays silently: an innocent change to one line shifts a result three functions away, and without tests nobody notices until the wrong number is in a report. With tests, the same mistake fails loudly in seconds, at your desk, before it costs anything.

02 · The assertion: a claim with teeth

The core tool is one statement: assert condition, "message" — silent when the condition holds, a named failure when it breaks. For engineering functions the pattern is the tolerance form from Lesson 2: assert abs(area_m2(0.1) - 0.007854) < 1e-6 — the hand-checked case (a 100 mm circle's \( \pi d^2/4 \)) pinned beside the function it certifies. Three or four assertions per function — a typical case, an edge (zero, the domain boundary), and a case exercising each branch — form a small permanent test suite that reruns in milliseconds. The habit converts "I checked it once when I wrote it" into "it is checked on every run", which is the difference between a memory and a guarantee.

03 · Exceptions and honest failure

Guard clauses (Lesson 3) raise exceptions at broken assumptions; testing meets them from the other side. A function fed garbage should fail LOUDLY — ValueError with a message naming the violated assumption — never return a plausible wrong number. Debugging itself is hypothesis testing, the scientific method at a keyboard: reproduce the failure with the smallest possible input, form a hypothesis about the earliest wrong value, place a print or assertion just before the suspect line, and bisect until the bug has nowhere to hide. Staring is not a method; bisection is.

04 · The verification working method

Adopt the platform's own pipeline at personal scale. Write the function; write its hand-checked case as an assertion immediately — while the paper calculation is still on the desk; add the edge-case and branch assertions; keep the whole block at the bottom of the file or in a test script, and run it after EVERY change. When a bug surfaces in use, its fix begins by writing the failing case as a new assertion — so the same bug can never return unannounced. And respect the honest limit: tests prove the presence of agreement with your checked cases, not the absence of all error — which is precisely why the cases are chosen with engineering judgment: the typical, the extreme, and the branch nobody believes will ever run. That judgment is the engineering; the assertions merely keep it enforced. A dozen well-chosen assertions guarding a personal library of checked functions is a small thing to maintain and a large thing to own: it is the difference between code you wrote and code you can vouch for.

Lesson 06 · Lists, dictionaries, and files

01 · Lists: ordered data with a name

Engineering data arrives in sequences — readings over a route, temperatures over a shift — and the list is Python's sequence: readings = [2.1, 4.5, 3.2], ordered, indexable, growable. Indexing starts at ZERO — readings[0] is the first item — and negative indices count from the end, readings[-1] being the latest reading, an idiom worth owning immediately. Slices carve subranges (readings[1:3] yields items 1 and 2 — the stop index is excluded, mirroring range), append grows the list in place, and the built-ins len, sum, min, max answer the first questions any dataset is asked. The list comprehension compresses the transform loop to a phrase — [x * 0.746 for x in hp_values] — and reads like the sentence it implements.

02 · Dictionaries: records with named fields

Where lists order items, dictionaries name them: pump = {"tag": "P-101", "power_kw": 11.5} is an equipment record whose fields are looked up by key — pump["power_kw"] — not by position. Adding a field is one assignment; asking whether one exists is "tag" in pump. The two structures compose into the universal data shape of plant work: a LIST of DICTIONARIES — one dict per equipment item, one list per fleet — which is exactly what a CSV file of records becomes the moment Python reads it.

03 · Files: where real data lives

The CSV — comma-separated values, one record per line — is the lingua franca of every historian export, test-bench log, and maintenance system dump. Reading one is three idioms stacked: iterate the file line by line, split(",") each line into fields, and convert numeric fields with float() at the border, exactly as Lesson 2 demanded. The with open(...) as f: form opens the file and guarantees it closes. Everything after the read is Lessons 3–4: loops, accumulators, and functions, now fed by real data instead of typed constants.

04 · The data-structure working method

Choose the structure by the question the data answers. Ordered measurements → list; named fields of one thing → dict; a fleet of records → list of dicts; and when you catch yourself tracking parallel lists (tags in one, powers in another), that is the signal the records want to be dicts. Convert types at the file border, never downstream. Trust the built-ins before writing loops — sum, max, len are tested code you did not have to write. And hand-trace list operations on a three-item example before applying them to thirty thousand rows, exactly as the loop lesson traced its accumulators: the small case proves the pattern, the large case merely repeats it.

One caution as datasets grow: a structure choice made casually at three records becomes architecture at thirty thousand. The list-of-dicts shape scales gracefully because every operation this course teaches — loops, comprehensions, filters, summaries — walks it unchanged; ad-hoc shapes (nested lists of mixed meaning, parallel lists) accumulate special cases until the script is mostly exceptions. Choose the boring, regular shape early and the analysis lessons ahead inherit clean footing.

Lesson 07 · Arrays and numerical work

01 · The array idea: operate on all of it

Lists process element by element through loops; numerical arrays (NumPy's ndarray) operate on whole datasets at once: kw = hp * 0.746 converts every horsepower value in one expression — vectorization. The gains are two: the code READS as the mathematics (one formula, visibly applied to everything), and it runs orders of magnitude faster, because the looping happens in optimized machinery below. For engineering data — a year of hourly readings is 8,760 numbers, a vibration waveform 65,536 — the array is the natural container, and the loop becomes the exception rather than the habit.

02 · The reductions: from many numbers to a few

Analysis usually ends in reductions: mean, std, min/max, sum — the descriptive statistics of a dataset in one call each. The standard deviation deserves engineering respect: it is the honest width of the scatter, the number that turns "about 4.25" into "4.25 ± 0.30", and the seed of every control chart the metrology course will draw. Alongside them, argmax answers WHERE — the index of the worst reading — which is often the more actionable question: not "how bad" but "which machine".

03 · Masks: asking questions of data

Comparison on an array yields an array of booleans — a mask — and the mask selects: readings[readings > 4.0] is every reading above the alarm line, in one expression: no loop, no if, no accumulator. Count them with sum of the mask, average them, or feed them onward. The idiom — condition, mask, selection — is the data-analysis sentence structure, and it compounds: masks combine with \( \texttt{\&} \) and \( \texttt{|} \) into compound questions ("above alarm AND after the overhaul") that still read as one line.

04 · The vectorized working method

Reach for the array whenever data is homogeneous and numeric; keep lists for mixed records. Write transformations as whole-array expressions and CHECK them on a three-value array first — vectorized code is compact enough that a wrong formula corrupts everything uniformly, so the small trace matters doubly. Prefer the built-in reductions to hand-rolled loops (they are tested code, and faster), and when a reduction's answer surprises, look at the mask counts before doubting the data: how many values, how many above threshold, how many NaN — the three questions that catch most data-quality issues at the door. Then the discipline that spans every lesson: units ride the variable names (vel_mm_s), conversions happen once, visibly, at the border, and any result consumed downstream gets an assertion pinning a known case. Arrays change the scale of what one line does; they change nothing about what makes the line trustworthy.

The payoff arrives immediately in the lessons ahead: plotting consumes arrays whole, curve fitting reduces them to parameters, and simulations generate them by the thousand. Vectorized thinking — state the operation once, apply it to everything, summarize with a reduction — is also precisely how plant historians and analysis platforms work internally, so fluency here reads directly onto the tools the later automation courses will operate.

Lesson 08 · Plotting engineering data

01 · The plot is an argument

An engineering plot exists to support a decision, and its anatomy is the argument's grammar: axes labeled WITH UNITS (an unlabeled axis is a number with no meaning — the chart version of the unit sins every course here prosecutes), a title that states what is shown, and a legend when more than one series appears. The choice of plot type is the choice of claim: a LINE plot claims continuity — readings ordered in time, trends; a SCATTER plot claims only "these pairs were observed" — right for correlation questions and unordered data; a BAR chart compares discrete categories — machines, months, failure modes. Choosing a line where a scatter belongs invents continuity the data never promised; the plot type is the first honesty decision.

02 · Scales that reveal

Linear axes hide what logarithmic axes reveal. Exponential decay — the cooling curves and filter-loading trends of the mathematics course — plots as a curve on linear axes but as a STRAIGHT LINE on a semilog (log-y) plot, whose slope reads the time constant: plotting \( \ln(T - T_\infty) \) against t was exactly this move, done by hand. Data spanning decades of magnitude — vibration spectra, particle counts — needs log scales simply to be seen: on a linear axis, the small values flatten into the floor. The working question before every plot: does this data live on one scale, or several?

03 · Sampling honesty

Plotted data was SAMPLED, and sampling has physics. A signal changing faster than half the sampling rate does not disappear — it ALIASES, reappearing as a false slow signal: a 50 Hz vibration sampled at 60 Hz shows up as a phantom 10 Hz wobble (\( |50 - 60| \)). The instrumentation course will formalize Nyquist; the plotting lesson's version is the caution: before trusting a surprising slow oscillation in sampled data, ask what the sampling rate was. Equally, decimating data for plotting speed (every tenth point) is legitimate — STATED; silently, it is how spikes vanish from trend reviews.

04 · The plotting working method

Build every plot in the same order. State the question the plot must answer; choose the type by the claim (trend → line, correlation → scatter, comparison → bar); choose scales by the data's structure (exponential suspicion → semilog; decades of range → log); label axes with quantities AND units; then read the plot yourself before showing anyone — does it actually answer the question, and would a skeptic's eye find the axis tricks (truncated baselines, cherry-picked windows) that make weak cases look strong? The mathematics course called the plot the test of first-order behavior; this lesson generalizes: the plot is where data confesses its structure, provided the plotter asks honestly. A figure that survives your own skepticism is ready for the morning meeting — and a figure that decided a question honestly is worth saving beside the script and data that produced it, completing the audit trail this course keeps building.

Lesson 09 · Curve fitting and interpolation

01 · Two questions a table cannot answer alone

Engineering data arrives as points: rows of a steam table, samples on a pump test, readings from a calibration bench. The questions arrive between the points — the head at a flow the test never held, the transmitter output at a pressure between bench settings. Two distinct operations close that gap. Interpolation reads between the rows of data you trust, adding no opinion of its own. Fitting proposes a model — a line, a low-order polynomial — and asks the data to vote on its parameters. Keeping the two distinct is the first discipline of this lesson: interpolation honors every point; fitting deliberately smooths over their scatter, and each is wrong where the other is right.

02 · Interpolation: reading between the rows

Between two table rows \( (x_1, y_1) \) and \( (x_2, y_2) \), linear interpolation draws the chord: \( y = y_1 + \frac{x - x_1}{x_2 - x_1}\,(y_2 - y_1) \). The fraction is just "how far across the interval x sits", and the same fraction is applied to the y-interval. Property tables are published dense precisely so that this chord is an excellent stand-in for the true curve; np.interp(x, xs, ys) does the row hunting and the arithmetic in one call, provided the sample points xs increase monotonically. The method's honest boundary is the table's edge: extrapolation — reading beyond the last row — is a different act entirely, because past the data the curve owes you nothing. Worse, np.interp does not protest at the edge: it silently returns the endpoint value, so the guard clause of Lesson 3 — assert the query lies inside the table's range — is part of every lookup that matters.

03 · Least squares: the best line through scatter

Calibration data never lies on a line; it scatters around one. For each candidate line, every point carries a residual — measured minus predicted, \( r_i = y_i - (m x_i + b) \). Least squares chooses the \( m, b \) that minimize \( \sum r_i^2 \), and for a straight line the answer is closed-form: \( m = \frac{n\sum x y - \sum x \sum y}{n \sum x^2 - (\sum x)^2} \), with \( b = \bar{y} - m\bar{x} \). np.polyfit(x, y, 1) evaluates exactly this. The degree argument is a temptation: a degree-9 polynomial threads all ten of your points exactly — residuals zero, \( R^2 = 1 \) — and swings wildly everywhere between them. Adding parameters always improves the numbers and, past the physics, always degrades the model; the fit stops summarizing the data and starts memorizing its noise.

04 · The fitting working method

The routine that keeps fits honest: PLOT the data before fitting anything — the eye catches curvature, outliers, and regime changes that summary numbers hide. Fit the simplest model the physics justifies, then plot the RESIDUALS: random scatter of roughly constant width says the model has taken all the structure the data offers; any pattern — a bow, a trend, a fan — is physics the model missed, an instruction to think, not to raise the degree. Distrust \( R^2 \) as a verdict: it measures variance explained, and a badly bowed fit can still post 0.99; the residual plot, not the score, is the honesty check. Report parameters with their units and alongside the data's domain, because the fit is a summary of THAT range — using it outside is extrapolation wearing a calculator's confidence. And when the goal is only to read between trusted rows, skip the model entirely: interpolate. The habits are small, but they are the difference between a calibration you can defend and a curve that merely looks professional. Both return in force when Metrology fits gauge data and Reliability fits life data — this lesson is their rehearsal.

Lesson 10 · Small simulations

01 · The equations engineers actually meet

Strip the plant to its rates and first-order differential equations appear everywhere: a tank fills at the rate flow-in minus flow-out; a motor housing cools at a rate proportional to how far it sits above ambient; receiver pressure decays as the leak bleeds it down. Each says \( dy/dt = f(y) \): the state's rate of change depends on the state itself. A few such equations have textbook solutions; most industrial variants — level-dependent outflow, staged inputs, valves that open mid-run — do not. Simulation sidesteps the impasse: instead of solving the equation, STEP it. March time forward in small increments, updating the state by its current rate, and the trajectory emerges numerically. The computer becomes a cheap laboratory — one where a tank can overflow a hundred times before breakfast and nothing needs mopping.

02 · Euler's method: the honest workhorse

The simplest march is Euler's: \( y_{n+1} = y_n + \Delta t \cdot f(y_n) \) — new state equals old state plus rate times step. It is nothing more than "keep the rate constant for a little while", the same approximation a hand calculation makes, automated. Its price is a per-step error that accumulates in proportion to \( \Delta t \): halve the step, roughly halve the error at a given time. That buys a wonderfully practical convergence test needing no theory at all — run the simulation, halve \( \Delta t \), run again; if the answer you report did not change at the precision you report it, the step is small enough. If it moved, halve again. The experiment IS the error analysis, and it costs seconds.

03 · When the step lies: stability

Take cooling toward ambient, \( dT/dt = -k\,(T - T_\infty) \). One Euler step multiplies the deviation from ambient by the factor \( (1 - k\,\Delta t) \). For small steps that factor sits just under 1 and the temperature relaxes exponentially, as physics says. But let \( k\,\Delta t \) exceed 2 and the factor drops below −1: each step OVERSHOOTS ambient farther than it started, flipping sign and growing — the simulated temperature oscillates with expanding amplitude while the real housing simply cools. The boundary \( \Delta t < 2/k \) is the stability limit, and it teaches the lesson's sharpest distinction: instability is not sloppiness, it is a qualitatively false solution. Between stable and accurate stands the time constant \( \tau = 1/k \) — the system's natural clock; steps around \( \tau/10 \) resolve the physics, steps near the limit merely avoid explosion.

04 · The simulation working method

Write the rate as a named function f(y) — Lesson 4's contract — so the model sits in one testable place and the marching loop never changes. Anchor the code before trusting it: the plain cooling case has an exact solution, so assert your loop reproduces it; the tank has a steady state you can compute by setting the rates equal, so run long and assert the level settles there. Only then vary toward the case with no formula — the varying inflow, the staged setpoint — because a loop that reproduces the known case has EARNED extrapolation to the unknown one. Run the halve-the-step experiment as routine, plot whole trajectories rather than trusting endpoints (Lesson 8: the plot is the argument), and keep the model's assumptions written beside the code. That last habit is the deepest: a converged, stable simulation is faithful to its EQUATIONS, and if the equations omit real physics — the heat the sun adds, the second leak — the laboratory lies with perfect numerical composure. Step-size error you can halve away; model error you can only think away. The distinction returns at full scale in Heat Transfer, System Dynamics, and every digital-twin conversation of the later automation courses.

Lesson 11 · Project: a maintenance data mini-analysis

01 · One project, the whole course

The course closes the way real analysis begins: a work-order export lands — one CSV row per maintenance job, with machine tag, failure date, hours down, hours to repair, a cause code — and someone asks what it says. The pipeline that answers is this lesson: load the raw file, clean it defensively, compute a handful of KPIs, plot the story, and write four honest sentences on top. Every prior lesson reports for duty: files and dictionaries carry the records in, arrays vectorize the arithmetic, assertions pin the steps, plots make the argument. The deliverable is not a number — it is a SCRIPT that turns the raw file into the numbers, rerunnable by anyone, next month, unchanged.

02 · Cleaning: where the analysis is won or lost

Real exports are dirty in boringly predictable ways: duplicated rows from double entry, a duration keyed in minutes into an hours column, negative times from swapped dates, blank cause codes. Cleaning is not clerical — every fix is an engineering decision, and the discipline is to make each one explicit in code: drop exact duplicates and COUNT what was dropped; range-check every numeric column and quarantine violators rather than silently deleting them; log rows in versus rows out. The payoff is quantitative: one mis-keyed duration can multiply a mean by ten while the median barely moves — which is why the robust statistics of Lesson 7 are cleaning's instruments, and why "how many rows survived, and why" is the first line of any defensible report.

03 · KPIs with honest denominators

Three ratios summarize a machine's month. MTBF — uptime divided by failure count — how long it runs between stoppages. MTTR — downtime divided by failure count — how long a stoppage lasts. Availability — uptime over total period — the fraction of the calendar the machine was there when needed. Each is one line of arithmetic and one buried decision: the denominator. Calendar hours or scheduled hours? Do planned overhauls count as downtime? Neither answer is wrong; the UNSTATED answer is. The other closing instrument is the Pareto view: rank downtime by cause and the top two or three causes typically own most of the loss — the ranking that converts a data file into next month's priorities. A KPI without its denominator stated, or a ranking without its period, is decoration.

04 · The reproducibility working method

The habits that make the analysis an instrument rather than a performance: the raw file is READ-ONLY — the script loads it, transforms in memory, writes results to new files, and never edits the source, because the untouched original plus the script IS the audit trail; rerunning must reproduce every figure exactly. One script runs raw to figures with no manual steps in between — no hand-pasted cells, no "then I fixed one value in the editor", since a single silent hand-edit breaks the chain that lets a reviewer trust the output. Assertions stand guard at the borders: row counts after each cleaning stage, ranges on every computed KPI. Figures carry their period and filters in the caption. And the four sentences on top state what was computed, over what denominator, with what excluded — the analyst's equivalent of showing units. This pattern — raw, script, assertions, figures, statement — returns at full strength in Reliability Engineering, where the same work-order data feeds Weibull fits and failure forecasting. This project is the rehearsal, and the habits are the deliverable.
